# Seata 功能介绍

Seata 是一个开源的分布式事务解决方案，核心目标是让一次跨服务、跨数据库的业务操作具备“要么都成功，要么都回滚”的一致性能力。

它主要解决这样一类问题：
- 订单服务已经写库成功
- 用户服务也已经更新成功
- 但订单服务后续又抛异常了
- 如果没有分布式事务，系统就会出现“部分成功、部分失败”的数据不一致

Seata 的价值，就是在这种跨服务场景下把多个本地事务组织成一个全局事务。

## 核心概念

### 1. TM（Transaction Manager，事务管理器）

#### 功能说明
TM 负责定义和发起全局事务，并决定最终是提交还是回滚。

#### 工作原理
- 当业务方法进入 `@GlobalTransactional` 时，TM 会向 Seata Server 申请一个全局事务 ID，也就是 `XID`
- 后续这个 `XID` 会沿着调用链传播到各个参与事务的服务
- 如果业务整体成功，TM 通知 TC 提交全局事务
- 如果中途任何一步失败，TM 通知 TC 回滚全局事务

#### 项目中的体现
```java
@GlobalTransactional(name = "create-order-tx", rollbackFor = Exception.class)
public Order createOrder(...) {
    ...
}
```

#### 使用场景
- 下单时同时写订单库和用户库
- 支付时同时写支付单、订单状态、积分账户
- 库存扣减 + 订单创建 + 优惠券核销

#### 生产适用性
✅ **适合生产环境**  
但是否采用 Seata，要看业务对强一致性的要求是否足够高。


### 2. TC（Transaction Coordinator，事务协调器）

#### 功能说明
TC 是 Seata Server，本身不执行业务 SQL，它负责记录全局事务状态、分支事务状态，并协调提交或回滚。

#### 工作原理
- 接收 TM 发起的全局事务注册请求
- 为全局事务分配 `XID`
- 接收各 RM 上报的分支事务注册请求
- 维护全局事务表、分支事务表、全局锁表
- 在全局提交时通知各分支完成提交
- 在全局回滚时通知各分支根据 `undo_log` 执行反向补偿

#### 项目中的体现
`docker-compose.yml` 已编排 `seata-server`：

```yaml
seata-server:
  image: seataio/seata-server:1.7.0
  environment:
    - STORE_MODE=db
```

Seata Server 元数据表脚本位于：
- `scripts/init-seata.sql`

核心表包括：
- `global_table`：记录全局事务
- `branch_table`：记录分支事务
- `lock_table`：记录全局锁

#### 使用场景
- 任何使用 Seata 的系统都必须依赖 TC

#### 生产适用性
✅ **适合生产环境**  
但需要独立运维、监控和高可用设计。


### 3. RM（Resource Manager，资源管理器）

#### 功能说明
RM 负责管理具体资源，也就是各个业务服务本地数据库中的事务分支。

#### 工作原理
- 在本地事务执行前后，RM 负责和 TC 通信
- RM 会为本地事务分支注册分支事务
- 在 AT 模式下，RM 会记录 SQL 执行前后的镜像信息
- RM 将回滚所需信息写入 `undo_log`
- 若全局事务失败，RM 根据 `undo_log` 执行反向 SQL 回滚

#### 项目中的体现
项目中订单服务、用户服务都引入了：

```xml
<dependency>
    <groupId>io.seata</groupId>
    <artifactId>seata-spring-boot-starter</artifactId>
</dependency>
```

同时业务库中都创建了 `undo_log` 表，例如：

```sql
CREATE TABLE IF NOT EXISTS undo_log (
    id BIGINT NOT NULL AUTO_INCREMENT,
    branch_id BIGINT NOT NULL,
    xid VARCHAR(100) NOT NULL,
    rollback_info LONGBLOB NOT NULL,
    ...
)
```

#### 使用场景
- 任何参与全局事务的业务库

#### 生产适用性
✅ **适合生产环境**


## Seata 常见模式

### 1. AT 模式

#### 功能说明
AT（Automatic Transaction）是 Seata 最常用的模式，基于关系型数据库本地事务和 `undo_log` 实现自动回滚，对业务代码侵入较低。

#### 工作原理
- 业务仍然执行本地 SQL，不需要开发者自己写补偿逻辑
- Seata 在执行更新类 SQL 时，会记录“前镜像”和“后镜像”
- 回滚时根据前镜像恢复原数据
- 同时利用全局锁防止脏写

#### 为什么叫“自动事务”
因为业务代码不用自己写 TCC 的 Try / Confirm / Cancel，也不用自己维护 Saga 状态机，只要：
- 引入 Seata
- 建好 `undo_log`
- 在全局入口加 `@GlobalTransactional`

就可以让本地数据库操作自动参与分布式事务。

#### 使用场景
- 订单、用户、库存、账户都使用关系型数据库
- 业务以增删改 SQL 为主
- 想降低改造成本

#### 生产适用性
✅ **非常常用**  
Seata 在大多数业务系统里，首先落地的通常就是 AT 模式。

#### 更优方案
- 对一致性要求极高、流程明确的核心交易链路，可考虑 TCC
- 对长事务、异步补偿流程，可考虑 Saga


### 2. TCC 模式

#### 功能说明
TCC（Try / Confirm / Cancel）要求业务自己定义三个阶段：
- Try：预留资源
- Confirm：正式提交
- Cancel：显式回滚

#### 工作原理
- Try 阶段不直接“最终提交”，而是先占用资源
- 如果全局事务成功，则执行 Confirm
- 如果失败，则执行 Cancel

#### 使用场景
- 核心支付、扣减余额、冻结库存等高价值链路
- 对事务语义要求极强，希望显式控制每一步

#### 生产适用性
✅ **适合生产环境**  
但开发复杂度明显高于 AT。

#### 更优方案
- 对当前项目这类以数据库更新为主的场景，AT 往往更合适


### 3. Saga 模式

#### 功能说明
Saga 面向长事务场景，将大事务拆成一系列本地事务，每一步失败时走对应补偿动作。

#### 工作原理
- 每一步本地事务成功后继续下一步
- 如果中间失败，不做数据库级别的即时回滚
- 而是按既定补偿链路反向补偿前面步骤

#### 使用场景
- 长流程审批
- 多系统异步编排
- 无法通过数据库事务快速锁定资源的业务

#### 生产适用性
✅ **适合特定生产场景**  
但通常不作为最先落地的模式。


## AT 模式实现原理详解

### 1. 全局事务是怎么开始的

当订单服务进入：

```java
@GlobalTransactional(name = "create-order-tx", rollbackFor = Exception.class)
public Order createOrder(...) { ... }
```

Seata 会做以下事情：

1. TM 向 TC 发起全局事务注册
2. TC 返回一个全局唯一的 `XID`
3. 当前线程上下文中绑定这个 `XID`
4. 后续本服务和远程调用链路都携带这个 `XID`

项目中可以看到日志打印：

```java
log.info("开始创建订单, XID: {}", RootContext.getXID());
```

这说明全局事务不是抽象概念，而是实际绑定到了当前业务线程。

### 2. 分支事务是怎么加入的

订单服务在 `createOrder()` 里会做几件事：
- 写订单主表
- 写订单项表
- 调用用户服务同步下单画像

其中：
- 订单服务本地数据库事务会注册为一个分支事务
- 用户服务收到带有 `XID` 的调用后，也会把自己的本地数据库事务注册为分支事务

此时，TC 维护的结构大致是：

```text
Global Transaction (XID=xxx)
├─ Branch 1: order-service / order_db
└─ Branch 2: user-service / user_db
```

### 3. `undo_log` 到底有什么用

`undo_log` 是 AT 模式的核心。

当 RM 执行更新类 SQL 时，Seata 会记录：
- SQL 执行前的数据快照（before image）
- SQL 执行后的数据快照（after image）

写入 `undo_log` 的核心目的不是“审计”，而是“回滚”。

如果全局事务失败，Seata 可以根据 `undo_log`：
- 找到这次分支事务改了哪些行
- 知道这些行修改前是什么值
- 然后生成反向更新，把数据恢复回去

可以把它理解成：
- 本地事务正常提交，保证单库事务成功
- `undo_log` 负责给未来可能发生的全局回滚预留“后悔药”

### 4. 全局锁为什么必要

如果没有全局锁，会出现一个危险场景：

1. 分支事务 A 修改了某行，但全局事务还没最终结束
2. 另一个无关事务 B 又改了同一行
3. 此时如果 A 失败回滚，就可能把 B 的结果覆盖掉，形成脏写

所以 Seata 会在 TC 的 `lock_table` 中记录全局锁，核心作用是：
- 在全局事务未结束时，保护已修改的数据行
- 防止其他事务在这个窗口内发生冲突更新

也就是说：
- 本地数据库锁保证单条 SQL 执行期间的一致性
- Seata 全局锁保证跨服务全局事务期间的一致性

### 5.1 Seata 全局锁的边界与误区

#### 误区 1：有了 Seata 全局锁，就不需要业务锁了

不是。

Seata 全局锁解决的是：
- 当前全局事务已经改过的那批数据，在全局事务结束前不要被其他受 Seata 管控的事务再次冲突写入

它不等于：
- 给整个业务对象加了一把“所有接口都必须遵守”的万能锁

换句话说，Seata 全局锁更像是：
- **分布式事务框架内部的并发保护机制**

而不是：
- **统一的业务互斥锁**

#### 误区 2：任何接口来改同一行，都会被 Seata 挡住

也不是。

Seata 全局锁要发挥完整效果，前提通常是对方也在 Seata 的事务体系内，例如：
- 对方也走 Seata AT 事务
- 或者对方显式参与 Seata 的全局锁检查

如果另一个接口只是普通本地事务：
- 没有 `XID`
- 没有参与 Seata 分支事务
- 没有检查全局锁

那么它仍然可能直接把数据改掉。

所以更准确的说法是：
- **Seata 全局锁能保护“它管得到的事务路径”**
- **管不到的普通写路径，仍然可能发生并发更新**

#### 误区 3：那是不是就一定会出现脏数据被覆盖

也不能简单这么理解。

Seata AT 模式在回滚时，会结合 `undo_log` 做脏数据校验。

也就是说：
- 如果 Seata 发现当前数据已经不是自己事务当初修改后的状态
- 说明中间有人又改过了这行数据
- 这时候它不会盲目把数据直接覆盖回旧值

它更可能表现为：
- 检测到脏写冲突
- 当前分支回滚失败
- 等待人工介入或后续补偿处理

所以风险不是“Seata 一定把别人覆盖掉”，而是：
- **如果系统里存在未统一治理的并发写路径，最终一致性和回滚成功率都会受影响**

#### 实际工程应该怎么做

Seata 全局锁通常只能解决“分布式事务框架层面”的一部分问题，真正上线时一般还要叠加以下手段：

- **业务分布式锁**
  - 例如按 `userId`、`orderId` 加 Redisson 锁
  - 解决同一个业务对象被多个接口同时修改的问题

- **数据库唯一约束 / 乐观锁**
  - 防止更新覆盖和重复写入
  - 作为最后一道数据正确性保障

- **接口幂等**
  - 防止重复请求造成重复事务

- **统一写入口**
  - 尽量不要让多个完全独立的接口随意修改同一批核心数据

#### 一句话总结

- `Seata 全局锁`：保护“Seata 事务中的已改数据”在事务完成前不被同类事务冲突写入
- `业务锁`：保护“同一个业务对象”不被不同接口并发修改
- `乐观锁/唯一约束`：保护最终落库结果不被错误覆盖

### 5. 全局提交是怎么完成的

当所有分支都成功后：

1. TM 通知 TC 提交全局事务
2. TC 将全局事务状态更新为提交中 / 已提交
3. 各分支本地事务其实已经提交完成
4. AT 模式下提交阶段主要是做收尾工作，例如异步删除 `undo_log`

注意：
- AT 模式的“提交”不是再执行一次业务 SQL
- 真正的数据写入已经在本地事务阶段完成
- 全局提交更像是“确认整个事务成功结束”

### 6. 全局回滚是怎么完成的

如果订单服务在用户侧同步成功后主动抛异常：

```java
if (simulateDistributedTxFailure) {
    throw BusinessException.serverError("模拟用户侧同步成功后订单服务异常，验证 Seata 全局回滚");
}
```

则过程会变成：

1. TM 感知到异常，通知 TC 回滚全局事务
2. TC 找到这个 `XID` 关联的全部分支事务
3. TC 依次通知各 RM 回滚自己的分支
4. RM 读取 `undo_log`
5. RM 根据 before image 把数据恢复成修改前状态
6. 订单库、用户库最终都回到事务开始前的状态

这就是 Seata 在项目里实现“跨服务一起回滚”的根本原理。


## 项目中的配置

### 1. 订单服务

```yaml
seata:
  enabled: ${SEATA_ENABLED:false}
  application-id: ${spring.application.name}
  tx-service-group: ${spring.application.name}-tx-group
  registry:
    type: nacos
    nacos:
      application: seata-server
```

说明：
- `enabled`：是否启用 Seata
- `application-id`：当前应用标识
- `tx-service-group`：事务分组名
- `registry/config`：告诉客户端去哪里找到 Seata Server 和配置中心

### 2. 用户服务

用户服务也有对应的 Seata 配置，并作为分支事务参与者接入。

### 3. Seata Server 元数据表

`scripts/init-seata.sql` 中初始化了：
- `global_table`
- `branch_table`
- `lock_table`

### 4. 业务库回滚日志表

`scripts/init-db0.sql`、`init-db1.sql`、`init-db2.sql`、`init-db3.sql` 中都补齐了：
- `undo_log`


## 项目中的业务案例

### 场景说明

本项目用“创建订单 + 同步用户下单画像”演示 Seata：

1. 订单服务创建订单
2. 订单服务远程调用用户服务
3. 用户服务更新 `orderCount`、`lastOrderNo`
4. 如果 `simulateDistributedTxFailure=true`，订单服务会在远程调用成功后主动抛异常

### 没有 Seata 会怎样

- 订单服务本地事务回滚
- 用户服务已经成功更新
- 最终出现跨服务数据不一致

### 开启 Seata 后会怎样

- 订单服务回滚
- 用户服务分支事务也回滚
- 订单库和用户库都回到操作前状态


## 使用场景

- 跨服务下单、扣库存、记账户
- 订单 + 用户画像同步
- 营销系统多服务联合写入
- 多库一致性要求较高的核心业务链路


## 生产适用性

✅ **适合生产环境**  
尤其适用于：
- 业务以关系型数据库为主
- 强一致性要求较高
- 不希望手写大量补偿逻辑

但也要注意：
- 会引入额外的事务协调成本
- 数据库会增加 `undo_log` 写入开销
- 高并发热点更新场景下，全局锁可能带来等待和冲突
- 长事务不适合用 AT 长时间占用资源


## 更优方案或替代思路

### 1. Seata AT
- 适合当前项目这类“数据库更新为主”的场景
- 改造成本低，是最现实的方案

### 2. TCC
- 一致性控制更强
- 更适合支付、余额、冻结类核心交易
- 但业务侵入高，开发成本大

### 3. Saga
- 更适合长流程、异步补偿
- 不追求数据库级即时回滚

### 4. 最终一致性消息方案
- 如本地消息表 + MQ + 幂等消费
- 更适合吞吐优先、允许短暂不一致的系统

### 5. 生产选型建议
- **中小型业务、以数据库事务为主**：优先 Seata AT
- **强资金交易**：优先 TCC
- **长链路异步业务**：优先 Saga 或消息最终一致性


## 各模式对比建议

| 模式 | 一致性强度 | 开发复杂度 | 业务侵入 | 适合场景 |
|------|------------|------------|----------|----------|
| AT | 较强 | 低 | 低 | 数据库更新型业务 |
| TCC | 很强 | 高 | 高 | 资金、库存冻结 |
| Saga | 最终一致 | 中 | 高 | 长流程异步编排 |


## 本项目最佳实践

本项目当前采用的是 **Seata AT 模式**，原因是：

1. 核心事务操作以数据库写入为主
2. 已有订单服务和用户服务两个明确分支
3. 希望以较低改造成本实现跨服务回滚
4. 可以通过 `simulateDistributedTxFailure=true` 直接验证回滚闭环

这使得项目中的分布式事务能力不仅停留在“引入依赖”，而是已经具备真实业务编排、真实回滚验证和完整基础设施支撑。
